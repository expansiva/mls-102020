- 2026-07-06: Fixed the "checkpoint shows as another clarification / never renders" bug. `checkpoint-draft` is now a no-LLM wrapper agent step (mirroring `agentNewSolutionFinal`) whose child clarification `checkpoint-draft-view` renders `widgetNsDraft`. A clarification placed as a flat sibling in `nextSteps` cannot be resolved by the frontend (`getInteractionStepId` walks `interaction.payload`), so it must be nested under an agent. `agentNsDraft.beforePromptStep` completes the wrapper without an LLM call; `e1-draft` no longer calls `activateCheckpointClarification` (the wrapper auto-runs via its dependency on `e1-draft`); `beforeClarificationStep` now renders on planId `checkpoint-draft-view`. `e2-journeys` now depends on `checkpoint-draft-view` (the step that completes on human approval).
- 2026-07-06: SUPERSEDED the wrapper approach — it deadlocked (verified in msgtask3: `checkpoint-draft` stuck `in_progress`). An agent step with a child clarification is a "container": the parent cannot complete until the child finishes, and the child (`dependsOn` the parent) cannot start until the parent completes. New approach matches the working `e1-clarification` / `agentNewSolution2Requirements` pattern: `checkpoint-draft` is a plain agent step that EMITS the review clarification into its OWN `interaction.payload` via `prompt_ready` (`checkpointSystemPrompt`, general); `afterPromptStep` keeps the payload (`handleCheckpointDraftPayload`); `beforeClarificationStep` renders `widgetNsDraft` on planId `checkpoint-draft`. On approve, `widgetNsDraft` writes a completed `checkpoint-draft-answer` result (which completes `checkpoint-draft` via children and unlocks `e2-journeys`, now depending on `checkpoint-draft-answer`). Rule: never use an agent step as a passive container for a child step.
- 2026-07-06: REMOVED the E1 draft human checkpoint entirely. Per PropostaAgentNewSolution §8, there are only TWO human interactions: (1) the opening clarification and (2) the journeys (E2). E1 now runs automatically: on a green gate `afterPromptStep` calls `approveNsStep(pipeline, 'e1-draft', 'auto')` and completes, so the pipeline continues straight to E2 (`buildPlannedTree`: `e2-journeys` now `dependsOn ['e1-draft']`; no `checkpoint-draft` step). Removed the `checkpoint-draft` branches from `beforePromptStep`/`afterPromptStep`/`beforeClarificationStep`, plus `checkpointSystemPrompt` and `handleCheckpointDraftPayload`. `widgetNsDraft.*` stays in the repo but is no longer wired (E1 has no human screen). The E1 blocking clarification (`e1-clarification-extra`) is unchanged — it belongs to interaction #1.
- 2026-07-18 — N5 (newSolution_10): the e1 clarification step (`applyInitialClarification`) now completes with cleaner `input_output`. Idempotent (already-deleted files are skipped) and opt-in (only on explicit /rebuild). Flag parsing in helpers/nsFastMode.parseNsRebuildMode/isNsRebuildMode (unit-tested); wired in agentNewSolution.beforePromptImplicit.
