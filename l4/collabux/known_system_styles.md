# Famílias de experiência conhecidas

## Objetivo

Permitir que o cliente diga que está acostumado com determinado tipo de sistema.

Exemplo:

```text
"Estamos acostumados com Salesforce"
```

ou:

```text
"Queremos algo mais parecido com Dynamics"
```

A plataforma deve transformar isso em preferência UX, sem copiar assets ou identidade visual proprietária.

## Estilos iniciais

### `salesforceStyle`

Bom para:

- CRM;
- cadastros com histórico;
- dashboards comerciais;
- record workspace.

Padrões:

- painel de registro;
- ações contextuais;
- abas/seções;
- cards de resumo;
- navegação orientada a entidade.

### `dynamicsStyle`

Bom para:

- backoffice;
- ERP/CRM;
- command bar;
- formulários administrativos;
- grids com ações.

Padrões:

- barra de comandos;
- lista + painel;
- ações agrupadas;
- formulários estruturados.

### `oracleStyle`

Bom para:

- ERP;
- operações administrativas densas;
- tabelas e filtros;
- telas financeiras/operacionais.

Padrões:

- alta densidade;
- filtros fortes;
- tabelas robustas;
- foco em eficiência operacional.

### `sapFioriStyle`

Bom para:

- processos empresariais;
- object pages;
- overview pages;
- guided flows.

Padrões:

- header informativo;
- seções em cards;
- object detail;
- fluxo por etapas.

### `squarePosStyle`

Bom para:

- POS;
- varejo;
- atendimento rápido;
- tablet/touch.

Padrões:

- botões grandes;
- catálogo visual;
- carrinho lateral;
- fluxo rápido.

### `jiraBoardStyle`

Bom para:

- filas;
- estados;
- kanban;
- operação por status.

Padrões:

- colunas;
- cards;
- ações rápidas de mudança de status;
- foco no trabalho em andamento.

### `powerBiStyle`

Bom para:

- dashboards;
- indicadores;
- análise;
- insights.

Padrões:

- KPIs;
- cards;
- filtros globais;
- gráficos/listas de insight.

## Observação

Esses estilos são famílias de interação e organização.

Eles não autorizam:

- uso de logos;
- cópia de assets;
- reprodução pixel-perfect;
- uso de CSS/HTML extraído;
- engenharia reversa.
